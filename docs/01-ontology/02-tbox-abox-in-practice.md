# TBox / ABox 분리의 실무적 의미

Ontology를 이야기하다 보면  
TBox / ABox라는 용어를 자주 만나게 됩니다.

이 문서에서는 이 개념을  
**이론 설명이 아니라, 실무에서 왜 중요한지** 중심으로 설명합니다.

---

## TBox와 ABox의 기본 개념

간단히 말하면 다음과 같습니다.

- **TBox (Terminological Box)**  
  → 도메인의 개념, 관계, 제약  
  → “무엇이 무엇인가”에 대한 기준

- **ABox (Assertional Box)**  
  → 실제 데이터 인스턴스  
  → “어떤 것이 존재하는가”에 대한 사실

이 둘의 차이는  
**변하지 말아야 할 것과, 계속 변하는 것의 분리**입니다.

---

## 실무에서 자주 겪는 실패 패턴

다음과 같은 상황은 매우 흔합니다.

- 데이터가 늘어날수록
- 프로젝트가 늘어날수록
- 사람이 바뀔수록

그래프의 의미가 점점 흐려집니다.

왜일까요?

대부분의 경우:
> **TBox와 ABox가 섞여 있기 때문**입니다.

---

## TBox와 ABox가 섞였을 때 생기는 문제

### 1. 개념이 데이터에 끌려다님
- “이번 데이터에는 이런 케이스가 있어서…”
- 그때그때 개념 정의가 바뀜

### 2. 같은 개념이 여러 형태로 존재
- 프로젝트별로 다른 노드 타입
- 관계 의미가 일관되지 않음

### 3. 그래프 확장이 어려워짐
- 새로운 도메인 추가 시 기존 구조 붕괴
- 쿼리는 되지만 의미적으로 신뢰 불가

이 상태의 그래프는:
- 커질수록 위험해지고
- 설명하기 어려워지며
- 결국 신뢰를 잃습니다.

---

## TBox / ABox 분리는 무엇을 해결하는가

TBox와 ABox를 명확히 나누면 다음이 달라집니다.

### 1. 기준이 고정된다
- 데이터가 바뀌어도
- 개념 정의는 흔들리지 않음

### 2. 데이터 품질을 검증할 수 있다
- 이 인스턴스가 이 타입에 속해도 되는가?
- 이 관계는 허용되는가?

### 3. 확장이 쉬워진다
- 새로운 데이터, 새로운 도메인 추가 시
- 기준 위에 얹기만 하면 됨

---

## 실무에서의 역할 분담

GoRAG Lab에서는 보통 다음과 같이 역할을 나눕니다.

- **TBox**
  - Ontology 정의
  - 허용 관계
  - 제약 조건
  - 타입 계층 구조

- **ABox**
  - 문서에서 추출된 엔티티
  - ERP/업무 시스템 데이터
  - 기간별 수치, 이벤트

이 분리가 명확할수록:
> KG는 “데이터 덩어리”가 아니라  
> **지식 구조**로 동작합니다.

---

## 요약

- TBox는 기준이고, ABox는 데이터다
- 이 둘을 섞으면 그래프는 빠르게 무너진다
- 분리하면 검증, 확장, 설명이 가능해진다
- 실무에서 TBox/ABox 분리는 선택이 아니라 필수다
