# LangGraph 기반 워크플로우 설계 관점

Agent가 여러 개라면  
이제 중요한 질문은 이것입니다.

> “이 Agent들을 어떤 순서와 조건으로 실행할 것인가?”

LangGraph는 이 문제를  
그래프 형태의 워크플로우로 해결합니다.

---

## 워크플로우는 코드가 아니다

워크플로우는 단순한 실행 순서가 아닙니다.

워크플로우는:
- 의사결정의 흐름
- 실패 시 분기
- 반복 조건
- 종료 기준

을 표현합니다.

즉:
> 워크플로우는 **추론의 구조**입니다.

---

## LangGraph의 적합성

LangGraph가 적합한 이유는 명확합니다.

- 상태(State)를 명시적으로 관리할 수 있고
- 분기와 반복이 자연스럽고
- Agent 간 책임 분리가 명확하기 때문입니다.

GoRAG Lab에서는  
LangGraph를 **Agent 실행 엔진**으로 사용합니다.

---

## 워크플로우 설계 원칙

- 상태는 최소화하되, 명확하게
- 각 노드는 하나의 책임만
- 실패 경로를 반드시 설계
- “성공”의 정의를 명확히

워크플로우가 복잡해질수록  
설계 문서의 중요성은 커집니다.

---

## 요약

- 워크플로우는 실행 순서가 아니다
- 추론 구조를 표현하는 도구다
- LangGraph는 Agent 시스템에 잘 맞는다

다음 챕터에서는  
이 구조가 **실제 문제에서 어떻게 쓰이는지**를 살펴봅니다.
