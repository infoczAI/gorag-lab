# Ontology는 왜 중요한가

Knowledge Graph를 처음 만들 때 가장 흔한 실수는  
“일단 노드와 관계부터 만들자”입니다.

하지만 Ontology 없이 만들어진 그래프는  
규모가 커질수록 빠르게 무너집니다.

---

## Ontology란 무엇인가 (실무 관점)

Ontology는 단순한 용어 사전이 아닙니다.

GoRAG Lab에서 Ontology는:
- 도메인에서 **무엇을 개념으로 볼 것인지**
- 어떤 관계만 허용되는지
- 어떤 속성이 어떤 타입을 가져야 하는지

를 정의하는 **제약과 기준의 집합**입니다.

즉 Ontology는:
> Knowledge Graph의 **기준면(TBox)** 역할을 합니다.

---

## Ontology가 없는 Graph의 문제

Ontology 없이 그래프를 확장하면 다음 문제가 반복됩니다.

- 같은 개념이 다른 이름으로 계속 생성됨
- 관계의 의미가 프로젝트마다 달라짐
- 데이터 적재 기준이 사람마다 다름
- 쿼리는 되지만, 의미적으로는 틀린 결과

이 상태의 그래프는:
- 유지보수가 어렵고
- 재사용이 불가능하며
- 설명력이 급격히 떨어집니다.

---

## TBox / ABox 분리의 의미

Ontology 기반 KG에서는 보통 다음을 구분합니다.

- **TBox (Terminological Box)**  
  → 개념, 관계, 제약 (변하지 않는 기준)

- **ABox (Assertional Box)**  
  → 실제 데이터 인스턴스 (계속 변하는 값)

이 분리가 중요한 이유는:
- 데이터가 바뀌어도
- 기준이 흔들리지 않기 때문입니다.

---

## Ontology는 문서가 아니라 제약이다

Ontology를 문서로만 관리하면 실패합니다.

Ontology는:
- 코드
- 그래프 스키마
- 검증 로직

과 연결되어 **실제로 강제**되어야 의미가 있습니다.

그렇지 않으면 Ontology는:
> “잘 정리된 설명서”로 끝납니다.

---

## 요약

- Ontology는 용어 정리가 아니다
- Knowledge Graph를 지탱하는 **제약 시스템**이다
- TBox/ABox 분리는 실무에서 매우 중요하다
