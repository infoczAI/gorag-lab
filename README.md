# GoRAG Lab

> Graph RAG & Ontology 기반 문서 지능 시스템을 실제로 구축하며 정리한 실무 플레이북

**GoRAG Lab**은  
Knowledge Graph, Ontology, Hybrid RAG(Vector + Graph), Agent 기반 추론을 활용해  
**대규모 문서 지능 시스템을 실제 서비스 환경에서 구축하며 얻은 설계 관점과 경험**을 정리한 공개 저장소입니다.

이 저장소는 튜토리얼이나 논문 요약이 아니라,  
**“실제로 만들다 보면 반드시 부딪히는 문제들”에 대한 사고 방식과 해결 전략**을 공유하는 것을 목표로 합니다.

---

## 이 저장소를 만든 이유

대부분의 RAG 예제는 다음 수준에서 멈춥니다.

- Vector Search
- Prompt Engineering
- 소규모 데모 파이프라인

하지만 실제 기업 환경에서는 곧바로 다음과 같은 문제를 만나게 됩니다.

- 문서마다 다른 용어 정의와 의미 충돌
- 스키마 없는 데이터로 인한 구조 붕괴
- 예외 조건, 제약, 암묵적 규칙의 누락
- “왜 이런 답이 나왔는지” 설명 불가

저희는 이런 문제들이 **모델 성능이 아니라 '모델링'의 문제**라고 판단했고,  
Ontology와 Knowledge Graph를 **RAG의 제어 계층(Control Layer)** 으로 사용해 왔습니다.

GoRAG Lab은 이 과정에서의 판단, 시행착오, 트레이드오프를 기록합니다.

---

## 목적

### 이 저장소는
- 실무 중심의 **설계 관점과 사고 기준**을 담고 있습니다
- Ontology / Graph / Agent를 언제, 왜 쓰는지 설명합니다
- 실제 프로젝트에서 유효했던 패턴과 실패 사례를 다룹니다
- 신규 팀원 온보딩을 위한 기술 레퍼런스로 활용됩니다

모든 예시는 **일반화·익명화·단순화**되어 있습니다.


## 이런 분들을 위한 저장소입니다

- **엔터프라이즈 RAG 시스템**을 실제로 구축 중인 엔지니어
- 문서 기반 시스템에서 **개념 일관성 문제**로 고생 중인 팀
- Graph RAG를 데모 수준이 아닌 **운영 관점**에서 보고 싶은 분
- GoRAG 프로젝트에 새로 합류한 팀원

빠른 결과를 원하는 분보다는,  
**왜 이런 구조가 필요한지 고민하는 분**께 더 잘 맞습니다.

---

## GoRAG Lab의 관점

저희는 다음을 중요하게 봅니다.

- Retrieval은 검색 문제가 아니라 **모델링 문제**입니다
- Ontology는 문서가 아니라 **제약 시스템**입니다
- Graph는 단독이 아니라 **LLM 추론과 결합될 때 의미가 있습니다**
- 엔터프라이즈 환경에서는 **정확도보다 설명 가능성**이 더 중요합니다

이 저장소의 모든 내용은 이 관점을 전제로 작성되어 있습니다.
