# GoRAG Lab

> Knowledge Graph & Ontology 기반 문제 해결을 실무에서 다루는 엔지니어링 플레이북

**GoRAG Lab**은  
Knowledge Graph(KG)와 Ontology를 기반으로,  
문서 분석, 추론, 검색, 의사결정 지원 등 **구조적 문제가 있는 영역을 실제로 해결하며 얻은 설계 관점과 경험**을 정리한 public repo입니다.

---

## Knowledge Graph(KG)란 무엇인가

Knowledge Graph는 단순한 “그래프 DB”가 아니며, 다음을 동시에 만족하는 **지식 표현 구조**입니다.

- 도메인 개념(Concept)이 명시적으로 정의되어 있고
- 개념 간 관계(Relation)가 의미적으로 제약되며
- 데이터 인스턴스가 그 위에 일관되게 적재되고
- 탐색, 추론, 설명이 가능한 상태

즉 KG는:
- 데이터 저장 방식이 아니라 **사고 구조**
- 검색 인프라가 아니라 **의사결정 인프라**
- 시각화 도구가 아니라 **제약과 규칙의 표현 체계**

입니다.

Ontology는 이 Knowledge Graph가 **무너지지 않도록 잡아주는 기준면(TBox)** 역할을 합니다.

---

## 이 저장소를 만든 이유

많은 문제들이 다음과 같은 형태를 띱니다.

- 데이터는 많은데, 구조가 없다
- 규칙은 있는데, 코드로만 흩어져 있다
- 계산은 가능한데, 왜 그런 결과가 나왔는지 설명이 안 된다
- 용어는 같은데, 맥락마다 의미가 다르다

이런 문제들은 단순히:
- SQL을 잘 짠다고
- 모델 성능을 올린다고
- Vector Search를 붙인다고

해결되지 않습니다.

저희는 이런 문제들의 공통 원인을  
**도메인 지식이 구조화되지 않은 상태로 흩어져 있기 때문**이라고 보고,  
KG와 Ontology를 통해 이를 구조화해 왔습니다.

---

## Graph RAG는 전체 중 일부입니다

최근 Graph RAG가 주목받고 있지만,  
이는 **Knowledge Graph를 활용하는 한 가지 응용 패턴**일 뿐입니다.

GoRAG Lab에서는 다음과 같은 문제들을 함께 다룹니다.

### 📄 문서 중심 문제
- (예) 계약서 / 약관 / 매뉴얼 분석
- 예외, 조건, 제약이 많은 문서의 구조화
- 문서 간 개념 불일치 해결

### 🧮 계산·분석 중심 문제
- (예) ERP 구조 기반 모델링
- 구성 요소별 비용 영향 분석
- “어떤 요인이 왜 결과에 영향을 줬는가” 설명 문제
- 수식 + 규칙 + 데이터가 결합된 추론

### 🔍 검색·탐색 문제
- 키워드가 아닌 **개념 기반 탐색**
- 관계를 따라가는 탐색 (원인 → 결과)
- 단순 Top-k 검색으로는 답이 안 나오는 질의

### 🤖 추론·에이전트 문제
- 규칙 기반 + LLM 추론의 결합
- 단계적 사고가 필요한 질의
- 중간 추론 결과를 설명해야 하는 경우

Graph RAG는 이 중 **문서 질의응답 영역에서의 적용 사례**로 다뤄집니다.

## 이 저장소는 무엇이고, 무엇이 아닙니다

### ✅ 이 저장소는
- Knowledge Graph를 **언제 쓰는지**보다 **왜 쓰는지**를 설명합니다
- Ontology를 설계할 때의 판단 기준을 공유합니다
- 도메인 모델링, 추론, 탐색의 연결 지점을 다룹니다
- 신규 팀원이 “우리는 왜 이렇게 설계하는가”를 이해하기 위한 레퍼런스입니다

### ❌ 이 저장소는
- Graph DB 튜토리얼이 아닙니다
- 특정 벤더(Neo4j 등) 사용법 설명서가 아닙니다
- 논문 요약 모음이 아닙니다
- 내부 시스템을 그대로 공개한 자료가 아닙니다

모든 예시는 **익명화·일반화**되어 있으며,  
사고 구조와 설계 논리를 중심으로 작성됩니다.

## 이런 분들을 위한 저장소입니다

- KG / Ontology를 **실제 문제 해결에 쓰고 싶은 엔지니어**
- RAG를 넘어 **구조적 추론 시스템**을 고민하는 팀
- 원가, 규칙, 문서, 도메인 지식이 복잡하게 얽힌 문제를 다루는 분
- GoRAG 프로젝트에 새로 합류한 팀원

“빠르게 데모를 만들고 싶은 분”보다는 **이 구조가 왜 필요한지 이해하려는 분**께 더 잘 맞습니다.

---

## GoRAG Lab의 기본 관점

- Knowledge Graph는 저장소가 아니라 **문제 분해 방식**입니다
- Ontology는 문서가 아니라 **제약과 의미의 기준**입니다
- Graph는 검색을 위해서가 아니라 **설명을 위해** 존재합니다
- 엔터프라이즈 문제에서 중요한 것은  
  *정답 하나*보다 **왜 그런 판단이 나왔는지**입니다

---

## 공개 범위에 대해

본 저장소에는 다음이 포함되지 않습니다.

- 실제 고객사 정보
- 내부 전체 스키마
- 비용·성능 수치
- 그대로 재현 가능한 운영 파이프라인

우리는 **결과물이 아니라 사고 과정을 공유**합니다.

## Claude Code에서 활용하기

이 저장소는 [MCP(Model Context Protocol)](https://modelcontextprotocol.io/) 설정이 포함되어 있어, Claude Code 사용 시 문서를 참조 자료로 활용할 수 있습니다.

### 사용 방법

1. 이 저장소를 clone합니다
2. 저장소 디렉토리에서 Claude Code를 실행합니다
3. MCP 서버 활성화를 승인합니다

이후 개발 중 다음과 같은 질문을 하면 문서를 참조하여 답변합니다:

- "온톨로지와 스키마의 차이가 뭐야?"
- "Graph RAG 설계할 때 주의할 점 알려줘"
- "TBox/ABox 분리가 왜 필요해?"

### 활용 시나리오

- **설계 검토**: 새로운 도메인 모델링 시 기존 패턴 참조
- **코드 리뷰**: Knowledge Graph 관련 코드 작성 시 설계 원칙 확인
- **온보딩**: 신규 팀원이 프로젝트 컨텍스트 파악
