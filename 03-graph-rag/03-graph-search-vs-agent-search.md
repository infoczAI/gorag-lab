# Graph 탐색과 Agent 기반 탐색의 차이

Graph RAG를 구현하다 보면  
다음 질문을 마주하게 됩니다.

> “그래프는 그냥 쿼리로 탐색하면 되지 않나요?”  
> “왜 Agent가 필요한가요?”

이 문서에서는  
Graph 탐색과 Agent 기반 탐색의 차이를 설명합니다.

---

## 전통적인 Graph 탐색

전통적인 Graph 탐색은 다음 특징을 가집니다.

- 정해진 패턴의 쿼리
- 명시적인 시작점과 종료 조건
- 예측 가능한 결과

이는 다음에 적합합니다.

- 반복적인 질의
- 구조가 안정된 문제
- 명확한 탐색 경로

---

## Agent 기반 탐색이 필요한 순간

다음 상황이 나타나면  
정적 쿼리만으로는 한계가 옵니다.

- 질의 의도가 모호하다
- 탐색 경로가 상황마다 달라진다
- 중간 결과에 따라 다음 행동을 바꿔야 한다
- 문서, 규칙, 데이터가 동시에 얽혀 있다

이때 탐색은:
> “쿼리 실행”이 아니라  
> **의사결정 과정**이 됩니다.

---

## Agent 기반 탐색의 역할

Agent는 다음 역할을 수행합니다.

- 질의를 분해한다
- 탐색 전략을 선택한다
- 중간 결과를 평가한다
- 다음 탐색 방향을 결정한다

Graph는:
- 탐색의 공간을 제공하고
- Ontology는 탐색의 제약을 제공합니다.

---

## Graph + Agent의 관계

중요한 점은 다음입니다.

- Agent가 Graph를 대체하지 않는다
- Graph가 Agent를 대체하지도 않는다

역할은 명확합니다.

- Graph: 지식과 구조
- Agent: 탐색과 판단
- LLM: 해석과 설명

Graph RAG는 이 세 가지의 **결합 패턴**입니다.

---

## 언제 Agent를 써야 하는가

다음 질문이 나오기 시작하면  
Agent 도입을 고려할 시점입니다.

- 이 질문은 정형 쿼리로 정의할 수 있는가?
- 탐색 경로가 항상 같은가?
- 중간 판단이 필요한가?

정형이면 쿼리로,  
비정형이면 Agent로 가는 것이 자연스럽습니다.

---

## 요약

- Graph 탐색은 정형 문제에 강하다
- Agent 기반 탐색은 비정형 추론에 강하다
- Graph RAG는 두 방식을 상황에 맞게 결합한다

다음 챕터에서는  
이 구조를 실제로 운영 가능한 시스템으로 만드는  
**Agent 설계와 워크플로우**를 다룹니다.
