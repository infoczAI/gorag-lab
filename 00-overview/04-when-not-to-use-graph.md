# 언제 Graph를 쓰지 말아야 하는가

GoRAG Lab은 Graph를 적극적으로 사용하지만,  
모든 문제에 Graph가 정답이라고 보지는 않습니다.

이 문서는 **Graph와 Knowledge Graph를 쓰지 말아야 할 경우**를 정리합니다.

---

## Graph가 과한 경우

다음과 같은 문제는 Graph가 과할 수 있습니다.

- 단순 집계 / 리포팅 중심 문제
- 관계가 거의 없는 평면 데이터
- 개념 정의보다 속도만 중요한 경우
- 구조가 거의 변하지 않는 정형 데이터

이 경우에는:
- RDB
- Columnar DB
- 단순 Vector Search

가 더 적절할 수 있습니다.

---

## Graph를 쓰면 실패하는 경우

다음 조건이 충족되지 않으면 Graph 도입은 실패 확률이 높습니다.

- 도메인 개념에 대한 합의가 없음
- Ontology를 유지할 조직/의지가 없음
- “일단 빨리 만들자”가 최우선 목표
- 그래프를 시각화 도구로만 인식함

Graph는:
> **초기 비용을 감수하고, 장기적 일관성을 얻는 선택**입니다.

---

## Graph가 진짜 필요한 신호

반대로, 다음 신호가 보이면 Graph를 고려할 시점입니다.

- 같은 질문에 사람마다 답이 다르다
- “왜 이런 결과가 나왔는지” 설명이 필요하다
- 규칙, 예외, 조건이 점점 늘어난다
- 키워드 검색으로 답이 안 나온다
- 데이터보다 **관계**가 더 중요하다

이때 Graph는 단순 저장소가 아니라  
**문제 구조를 드러내는 도구**가 됩니다.

---

## 요약

- Graph는 만능이 아니다
- 단순한 문제에는 단순한 도구가 낫다
- 하지만 구조적 복잡성이 쌓이기 시작하면  
  Graph와 Ontology는 강력한 선택지가 된다

이 기준 위에서 **Ontology 설계와 KG 활용을 어떻게 하는지**를 다룹니다.
