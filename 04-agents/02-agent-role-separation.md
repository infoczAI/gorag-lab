# Agent 역할 분리 전략

Agent를 도입할 때 가장 흔한 실수는  
“하나의 Agent가 다 하게 만드는 것”입니다.

이 경우 시스템은 빠르게 복잡해지고,  
디버깅과 확장이 어려워집니다.

---

## 역할 분리의 필요성

복잡한 문제는 보통 다음 단계를 포함합니다.

1. 질문 이해
2. 탐색 전략 수립
3. 지식 탐색
4. 중간 결과 평가
5. 최종 설명 생성

이 모든 단계를 하나의 Agent가 수행하면:
- 책임이 불명확해지고
- 판단 기준이 섞이며
- 결과를 신뢰하기 어려워집니다.

---

## GoRAG Lab의 기본 역할 분리

GoRAG Lab에서는 보통 다음과 같이 나눕니다.

- **Query Interpreter**
  - 사용자 질문 해석
  - 핵심 개념 추출

- **Planner Agent**
  - 탐색 전략 결정
  - 사용할 도구/경로 선택

- **Graph Explorer**
  - Knowledge Graph 탐색
  - 제약 및 규칙 적용

- **Verifier**
  - 결과 검증
  - 규칙 위반 여부 확인

- **Answer Composer**
  - 결과 요약
  - 설명 생성

각 Agent는:
> 하나의 책임만 가집니다.

---

## 역할 분리가 주는 효과

- 테스트가 쉬워진다
- 실패 원인을 추적할 수 있다
- 부분 교체가 가능해진다
- 사람이 개입할 지점이 명확해진다

Agent는 많아질수록 좋은 게 아니라,  
**역할이 명확할수록 강력**해집니다.

---

## 요약

- Agent는 기능이 아니라 역할로 나눈다
- 하나의 Agent는 하나의 책임만 가진다
- 분리는 복잡도를 낮추기 위한 전략이다

다음 문서에서는  
이 Agent들을 **워크플로우로 묶는 방법**을 다룹니다.
